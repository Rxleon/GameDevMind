# 热更新踩坑实录：旧资源残留导致「幽灵 UI」

> 一句话总结：回滚热更版本只换了 manifest，没清 CDN 边缘缓存和客户端下载目录，玩家界面叠了两套按钮，点哪都不对。

---

## 背景

项目是一款 Unity + Lua 的卡牌手游，热更链路是：打包 AssetBundle + Lua 脚本 → 上传 CDN → 客户端对比 manifest 增量下载。

某次活动紧急修复了一个 UI 布局 bug（`ui_shop.prefab` 和 `shop.lua`），热更发布 2 小时后运营反馈：**部分玩家商店界面出现两套重叠按钮**，点击事件错乱，甚至能点到已下架的商品入口。

---

## 症状

| 表现 | 比例 |
|------|------|
| 商店 UI 按钮重叠、文字重影 | 约 8% 活跃玩家 |
| 点击「购买」无反应或跳到错误商品 | 同上 |
| 完全正常 | 其余玩家 |

关键线索：**出问题玩家大多在热更发布前已打开过游戏，且未杀进程重启**；全新安装或清缓存后正常。

---

## 排查过程

### 第 1 步：怀疑 Lua 逻辑错误

对比 `shop.lua` 新旧 diff，只是改了布局坐标，没有 duplicate 创建 UI 的代码。

**结论：脚本逻辑不是主因。**

### 第 2 步：检查 AssetBundle 加载

在客户端加日志，发现同一帧内 `ui_shop.prefab` 被加载了两次，路径不同：

```
LoadAsset: persistent://hotupdate/ui_shop.ab  (v1.2.0)
LoadAsset: persistent://hotupdate/_bak/ui_shop.ab  (v1.1.9 残留)
```

原来下载新版本时，旧文件被移动到 `_bak` 目录「备回滚」，但 UI 框架的 AssetBundle 缓存字典**仍持有旧 bundle 引用**，新 bundle 加载后两套 Prefab 同时实例化。

### 第 3 步：CDN 与回滚链路

进一步查发布记录：第一次热更 MD5 写错，运维 10 分钟内回滚 manifest 到 `1.1.9`，但 CDN 上 `1.2.0` 的文件已预热，部分边缘节点仍返回新包；客户端 manifest 是旧的、文件是新的，MD5 校验失败后反复重试，留下半套 `_bak` 文件。

**结论：manifest 回滚 + 文件残留 + AB 缓存未失效，三重叠加。**

---

## 根因分析

1. **下载目录未做版本隔离**：所有热更文件平铺在 `hotupdate/`，回滚时只改 manifest 不删文件
2. **AssetBundle 缓存无版本键**：同一 logical path 对应多个物理文件时，缓存命中旧 bundle
3. **紧急回滚流程不完整**：只回滚 manifest，未触发 CDN 刷新与客户端强制清缓存策略

---

## 解决方案

| 改动 | 说明 |
|------|------|
| **版本目录隔离** | 热更文件写入 `hotupdate/{version}/`，切换版本时整目录替换 |
| **AB 缓存加版本后缀** | cache key = `{version}:{logical_path}` |
| **回滚 checklist** | manifest 回滚 + CDN purge + 可选「强制重下」标记 |
| **启动时校验** | 对比 manifest 与本地文件 MD5，不一致则清空当前版本目录重下 |

改后同类问题归零；紧急回滚从 30 分钟缩短到 8 分钟（含 CDN purge 自动化）。

---

## 效果

| 指标 | 改前 | 改后 |
|------|------|------|
| 热更相关 UI 错乱工单 | 周均 3～5 单 | 0 |
| 回滚可操作时间 | ~30 min（人工） | ~8 min（脚本） |
| 客户端热更失败重试率 | 2.1% | 0.3% |

---

## 经验教训

- manifest 是「目录」，不是「魔法」——文件级一致性与目录隔离同样重要
- 热更系统要设计**失败与回滚**路径，不能只测 happy path
- 发布 checklist：MD5 预检 → 灰度渠道 → 全量 → 监控下载失败率

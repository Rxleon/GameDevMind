# 支付踩坑实录：掉单与重复发货

> 一句话总结：客户端以「支付 SDK 回调成功」为发货依据，没做服务端验单 + 幂等，同一笔订单被发了三次道具。

---

## 背景

一款 IAP 手游，Unity 客户端接 Apple / Google 支付 SDK，服务端 Go + MySQL。上线初期日流水不高，直到某次 648 档礼包促销，客服收到玩家投诉：**付了一次钱，背包里多了 3 份礼包**。

更严重的是，后台对账发现还有一批**付了钱没收到货**的「掉单」。

---

## 症状

| 类型 | 表现 |
|------|------|
| **重复发货** | 同一 `transaction_id` 对应多条发货日志 |
| **掉单** | 支付平台显示成功，游戏内无道具，无发货记录 |
| **时序** | 重复发货多发生在弱网或玩家快速切后台时 |

---

## 排查过程

### 第 1 步：查重复发货日志

```sql
SELECT transaction_id, COUNT(*) FROM iap_delivery GROUP BY transaction_id HAVING COUNT(*) > 1;
```

同一 Apple `transaction_id` 最多出现 3 条记录，时间间隔 200ms～2s。

客户端代码逻辑是：

```
OnPurchaseSuccess(sku) → 直接 POST /api/iap/deliver { sku, receipt }
```

**没有等服务端验单结果**，且玩家连点、SDK 重复回调都会触发多次 POST。

### 第 2 步：查掉单

掉单玩家客户端有 `OnPurchaseSuccess` 日志，但服务端无对应请求——弱网下 POST 超时，客户端以为成功已展示 UI，实际服务端未收到。

### 第 3 步：对照平台文档

Apple / Google 均要求：**以服务端向平台验单结果为唯一发货依据**，客户端 receipt 不可信。

---

## 根因分析

1. **发货触发点在客户端**，缺少服务端验单闭环
2. **无幂等键**：`/deliver` 未按 `transaction_id` 去重
3. **无补单机制**：掉单玩家无法自动恢复

---

## 解决方案

```
客户端                         服务端                    支付平台
   │  receipt/token  ────────→  │  verify ─────────────→  │
   │                            │  ←── 验单结果 ──────────  │
   │  ←── 发货结果（幂等）──────  │  INSERT ... ON DUPLICATE │
```

| 改动 | 说明 |
|------|------|
| **服务端验单** | Apple App Store Server API / Google Play Developer API |
| **幂等表** | `iap_orders(transaction_id UNIQUE, status, ...)` |
| **客户端只展示「处理中」** | 发货 UI 以服务端推送或轮询结果为准 |
| **补单任务** | 定时拉取 pending 订单重试验单 |

改后重复发货归零，掉单率从 0.8% 降到 0.02%（主要为平台延迟）。

---

## 效果

| 指标 | 改前 | 改后 |
|------|------|------|
| 重复发货工单 | 促销日 40+ | 0 |
| 掉单率 | 0.8% | 0.02% |
| 对账差异 | 日均 ¥2万+ | 接近 0 |

---

## 经验教训

- **永远服务端验单**，客户端 success 回调只表示「SDK 认为成功了」
- 支付接口必须**幂等**，`transaction_id` 是唯一业务键
- 要有**补单与人工审核**兜底，不要赌网络永远正常
